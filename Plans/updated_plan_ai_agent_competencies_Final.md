# Итоговый план создания HTML-продукта оценки компетенций и навыков

## 1. Цель продукта

Создать локальный HTML-продукт в виде одного HTML-файла со встроенными CSS и JavaScript, который позволяет проводить оценку одного респондента по набору компетенций и навыков в рамках одной сессии. Пользователь продукта должен иметь возможность выбрать компетенцию и навык, отметить релевантные пункты «Не выполнил», получить автоматически рассчитанную оценку, ввести комментарии и ручную оценку, а затем сформировать итоговую таблицу и открыть Outlook с подготовленным письмом.

Продукт должен быть удобен для дальнейших визуальных и логических доработок после первого рабочего прототипа.

## 2. Общая продуктовая логика

Продукт представляет собой локальное мини-веб-приложение, которое запускается без сервера, открытием HTML-файла в браузере. Все данные о компетенциях, навыках, пунктах «Не выполнил», штрафных баллах, комментариях и оценках хранятся внутри страницы в JavaScript-структурах.

Продукт должен поддерживать сценарий последовательной оценки: пользователь переходит между компетенциями и навыками, заполняет данные, видит пересчет оценок и в конце получает итоговую форму для отправки результатов по электронной почте через Outlook.

## 3. Как продукт должен выглядеть

### 3.1. Общая композиция интерфейса

Интерфейс должен состоять из одного экрана и иметь следующую структуру:

- Сверху — поле «Введите ФИО респондента».
- Слева — навигационная область «Компетенции» и «Навыки».
- Слева под «Навыки» — раздел «Неофициальные комментарии» с 6 обязательными вопросами.
- По центру — блок «Не выполнил» и блок «Комментарии по ответам».
- Справа — блок «Итоговая оценка авто» и блок «Итоговая оценка ручной ввод».
- Внизу — кнопка «Оценка завершена».

### 3.2. Визуальные требования

Интерфейс должен быть выполнен:

- в светлом стиле;
- со сглаживанием и аккуратной современной типографикой;
- в эстетике, близкой к MacOS / Apple;
- с понятным визуальным разделением всех зон;
- с мягкими фонами, легкими тенями, скруглениями и аккуратными отступами;
- с возможностью последующих правок после первого визуального отображения.

### 3.3. Дополнительные требования к внешнему виду

- Каждый раздел должен быть четко виден и отделен от соседнего.
- Активная компетенция и активный навык должны быть явно выделены.
- Центральная зона должна быть самой удобной для чтения длинных списков и ввода комментариев.
- Правая зона должна визуально подчеркивать значения оценок.
- Цветовая палитра должна быть светлой, спокойной и не перегруженной.

## 4. Как продукт должен работать глазами пользователя

1. Пользователь открывает HTML-файл в браузере.
2. Пользователь видит экран с разделами, указанными в требованиях к внешнему виду.
3. Пользователь вводит ФИО в поле «Введите ФИО респондента».
4. Пользователь выбирает слева компетенцию.
5. Пользователь выбирает внутри компетенции конкретный навык.
6. Пользователь видит список «Не выполнил» по выбранному навыку.
7. Пользователь ставит галочки перед теми пунктами «Не выполнил», которые считает нужными.
8. После каждой галочки автоматически пересчитывается «Итоговая оценка авто» по выбранному навыку.
9. Пользователь может писать комментарии в блок «Комментарии по ответам».
10. Пользователь может вручную вводить значение в блок «Итоговая оценка ручной ввод».
11. Пользователь может в любой момент выбрать другой навык и увидеть уже его собственные данные: свои галочки, свой комментарий, свою автооценку и свою ручную оценку.
12. Пользователь заполняет 6 полей раздела «Неофициальные комментарии».
13. Если хотя бы одно из 6 полей не заполнено, кнопка «Оценка завершена» не открывает итоговую форму и показывает ошибки рядом с пустыми полями.
14. После нажатия кнопки «Оценка завершена» должна открываться итоговая форма со строками по оценке.
15. Под основной таблицей итоговой формы должна отображаться таблица «Неофициальные комментарии» с 6 ответами.
16. Под итоговой формой должна быть кнопка «Отправить», которая открывает Outlook с подготовленным письмом.

## 5. Логика данных и расчета

### 5.1. Основная логика оценки

Для каждого навыка начальная автоматическая оценка равна 6.

Каждый отмеченный пункт «Не выполнил» уменьшает эту оценку на свое значение штрафа.

Формула автоматической оценки по навыку:

`Итоговая оценка авто = 6 - сумма штрафов всех отмеченных пунктов «Не выполнил» по выбранному навыку`.

Все числовые оценки в продукте должны нормализоваться в диапазон 0..6:

- если значение больше 6, использовать 6;
- если значение меньше 0, использовать 0.

### 5.2. Правила хранения данных

Для каждого навыка должны независимо храниться:

- набор отмеченных пунктов «Не выполнил»;
- текущая автоматически рассчитанная оценка;
- ручная оценка;
- комментарий пользователя.

Переключение между навыками не должно приводить к потере введенных данных.

Ручная оценка является вспомогательным полем интервьюера и не используется в расчете итогового балла отчета.

### 5.3. Итоговая форма

После нажатия кнопки «Оценка завершена» должна собираться форма со строками, содержащими:

- Тема
- Навык
- Уровень
- Комментарии

Комментарий по навыку может отображаться отдельно дополнительным полем и не заменяет колонку «Комментарии».

В итоговый отчет включаются только навыки, в которых отмечен минимум один пункт «Не выполнил».

В колонке «Уровень» отображается формат: «Итоговая оценка авто - текстовый уровень».

Критерии текстового уровня:

- 0 — «Навык отсутствует»;
- 1 или 2 — «Начальный уровень»;
- 3 или 4 — «Базовый уровень»;
- 5 или 6 — «Продвинутый уровень».

Под основной таблицей итоговой формы обязательно выводится вторая таблица «Неофициальные комментарии» с 6 ответами в исходном порядке.

В колонке «Уровень» всегда отображаются числовая автооценка и текстовый уровень, рассчитанные только по «Итоговой оценке авто».

### 5.4. Outlook

Кнопка «Отправить» не отправляет письмо автоматически через сервер. Вместо этого она должна:

- собрать данные итоговой формы;
- превратить их в текст письма;
- открыть Outlook или почтовый клиент по умолчанию через `mailto:`;
- подставить тему письма и тело письма с результатами.

Технические требования к формированию письма:

- `subject` и `body` должны кодироваться через `encodeURIComponent`;
- переносы строк в `body` должны передаваться корректно;
- при невозможности сформировать рабочую `mailto`-ссылку из-за длины нужно показать готовый текст отчета для ручного копирования в письмо.

### 5.5. Неофициальные комментарии

Раздел «Неофициальные комментарии» содержит 6 обязательных вопросов с ответами:

1. В общем уровень начальный/базовый/продвинутый
2. Как отвечает на вопросы в глубину
3. Ответы структурные или поток сознания
4. Отношение там где не знает или ошибается
5. Делал с ИИ/без ИИ. Смог или нет подтвердить знания
6. Чем особо запомнился/Дополнительная информация

### 7.3. Минимальный набор функций реализации

- `clampScore(value)` — нормализация значения в диапазон 0..6;
- `calculateAutoScore(skill)` — расчет «Итоговой оценки авто» с учетом штрафов и нормализации;
- `buildSummaryRows(competencies)` — сбор итоговых строк только по навыкам с отмеченными пунктами «Не выполнил»;
- `getLevelLabel(autoScore)` — определение текстового уровня по шкале 0..6;
- `validateUnofficialComments(comments)` — проверка заполнения 6 обязательных неофициальных комментариев;
- `buildMailtoLink(summaryRows, respondentFullName)` — сбор темы и тела письма с корректной кодировкой.

## 8. Валидация

Перед формированием итоговой формы необходимо проверить:

- заполнено ли ФИО респондента;
- корректно ли введена ручная оценка (допустимый диапазон 0..6);
- заполнены ли все 6 полей раздела «Неофициальные комментарии»;
- корректно ли отображаются и сохраняются данные выбранного навыка при переключении между навыками.

Ошибки должны отображаться мягко и понятно, рядом с полями, без агрессивных alert-окошек как основного сценария.

## 9. Ограничения и запреты

При реализации продукта нельзя:

- менять названия компетенций;
- менять названия навыков;
- менять формулировки пунктов «Не выполнил»;
- менять значения штрафов;
- менять стартовую автоматическую оценку 6;
- придумывать новые сущности вместо заданной логики;
- реализовывать backend вместо локального HTML-решения;
- делать полностью автоматическую отправку письма без открытия Outlook или почтового клиента по умолчанию.

## 11. Критерии готовности продукта

Продукт считается готовым на базовом уровне, если:

- HTML-файл открывается локально без ошибок;
- интерфейс визуально соответствует требованиям;
- можно ввести ФИО;
- можно выбрать компетенцию и навык;
- по каждому навыку отображается свой список «Не выполнил»;
- блок «Неофициальные комментарии» из 6 полей отображается и валидируется как обязательный;
- галочки корректно влияют на итоговую оценку авто;
- комментарии и ручная оценка сохраняются отдельно по навыкам;
- итоговая форма формируется в структуре «Тема — Навык — Уровень — Комментарии»;
- в колонке «Уровень» показывается формат «оценка - текстовый уровень»;
- под основной таблицей отображается таблица «Неофициальные комментарии»;
- кнопка «Отправить» открывает Outlook или почтовый клиент по умолчанию с подготовленным письмом.
